Data pipeline · process map · ZCode

How data flows into Postgres and Redis — and who reads it back out.

Two clocks drive StockScreenClaude. Scheduled and bootstrap processes fetch from external providers and write durable rows to Postgres plus hot payloads to Redis. User-facing processes then read from those stores: scans stay cache-only, APIs return persisted data, and MCP / static / assistant surfaces reuse the same published artifacts. The diagram below traces that flow left to right.

01. End-to-end flow diagram

External data sources (left) are pulled by Celery worker processes (center-left) which write durable truth to Postgres and hot state to Redis (center). Read processes (center-right) pull from those stores and feed the external surfaces (right). The dashed line marks the cache-miss fallback path.

Live app pages

FastAPI + React via TanStack Query.

  • Scan: reads scans/scan_results or feature-store-backed pages.
  • Stocks: cached price history + fundamentals, Postgres fallback.
  • Breadth / Groups / Themes: read durable tables + UI snapshots.

On-demand computation

Celery user-scan workers.

  • Exact match: reads a published feature_run.
  • Compile path: queries stock_feature_daily with compiled filters.
  • Async path: cache-only price/fundamentals, writes scan_results.

Static & agent surfaces

Publish and inspect, no source-of-truth writes.

  • Static export: published artifacts + cached chart data → JSON.
  • MCP copilot: reads Postgres facts with citations.
  • Assistant: persists conversations, calls market tools.
Mental model: scheduled / background processes pay the cost of slow provider calls and durable writes. User-facing reads then stay fast — Postgres for truth, Redis for hot payloads and coordination, static bundles for read-mostly pages.

04. Schedule & worker topology

Celery Beat builds a dynamic per-market crontab when cache_warmup_enabled. Queue routing sends external fetches to data_fetch_*, compute to market_jobs_*, and manual scans to user_scans_*. Each market runs on its own worker, in parallel.
